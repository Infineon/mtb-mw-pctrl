MISRA-C:2012 Compliance

This section describes MISRA-C:2012 compliance and deviations for the Power Control Middleware Library.

MISRA stands for Motor Industry Software Reliability Association. The MISRA specification covers a set of 10 mandatory rules, 110 required rules and 39 advisory rules that apply to firmware design and has been put together by the Automotive Industry to enhance the quality and robustness of the firmware code embedded in automotive devices.

The MISRA specification defines two categories of deviations (see section 5.4 of the MISRA-C:2012 specification):

  • Project Deviations - deviations that are applicable for particular class of circumstances.
  • Specific Deviations - deviations that are applicable for a single instance in a single file.

Project Deviations are documented in current section below.

Specific deviations are documented in the source code, close to the deviation occurrence. For each deviation a special macro identifies the relevant rule or directive number, and reason.

Project Deviations

The following project-wide deviations from MISRA-C:2012 are documented. These deviations are applicable across the Power Control Middleware Library and are recorded in the Coverity deviation configuration file.

MISRA Rule Rule Description Description of Deviation(s)
Directive 4.2 All usage of assembly language should be documented. Disabled in the analysis configuration. The Power Control Middleware Library does not use inline assembly.
Directive 4.4 Sections of code should not be "commented out". Allow commenting out code sections. Commented-out code is used for documenting alternative configurations and is maintained alongside active code.
Directive 4.8 If a pointer to a structure or union is never dereferenced within a translation unit, then the implementation of the object should be hidden. Deviated since structure typedefs might be included from an external library (e.g., PDL, CMSIS) that is not under the project's control.
Directive 4.9 A function should be used in preference to a function-like macro. Function-like macros are used to allow more efficient code. The library uses macros for variant dispatch, memory section placement, and compiler attribute abstraction.
Rule 1.2 Language extensions should not be used. The Power Control Middleware Library supports the ISO:C99 standard. Compiler-specific extensions are used for memory section placement and function inlining, isolated in mtb_mw_pctrl_compiler.h behind compiler detection macros.
Rule 2.5 A project should not contain unused macro declarations. The Power Control Middleware Library provides a configurable API. Configuration macros define defaults that may not be used if the user overrides them in mtb_mw_pctrl_user_config.h. This is by design for the two-tier configuration architecture.
Rule 3.1 The character sequences
/* and //
shall not be used within a comment. 		Special comment symbols are required for Doxygen documentation support. This does not have any functional impact.
Rule 5.5 Identifiers shall be distinct from macro names. This rule applies to ISO:C90 standard. The Power Control Middleware Library conforms to ISO:C99 that does not require this limitation.
Rule 5.8 Identifiers that define objects or functions with external linkage shall be unique. During code analysis, the same source files are compiled multiple times with device-specific options. All object and function identifiers are actually unique for each specific run.
Rule 8.7 Functions and objects should not be defined with external linkage if they are referenced in only one translation unit. The Power Control Middleware Library provides API to the application. Functions and objects with external linkage are part of the public API, which is defined for application-level use only.
Rule 8.13 A pointer should point to a const-qualified type whenever possible. The const qualifier is not enforced on pointer parameters because a single API function signature is shared across multiple variant implementations, where one variant may modify the pointed-to data and another may not.
Rule 11.4 A conversion should not be performed between a pointer to object and an integer type. The cast from unsigned int to pointer does not have any unintended effect, as it is a consequence of the definition of a structure based on hardware registers.
Rule 15.5 A function should have a single point of exit at the end. Early returns are used for parameter validation guard clauses, which improves readability. This rule has been disapplied in MISRA-C:2025.

Specific Deviations

The following specific deviations apply to individual files in the Power Control Middleware Library. These findings were identified during Coverity static analysis and are documented here with rationale.

MISRA Rule Rule Description File(s) Description of Deviation(s)
Directive 4.5 Identifiers in the same name space with overlapping visibility should be typographically unambiguous. cy_pctrl_log.h The logging subsystem uses macro names that are typographically similar by design, following a consistent naming convention (e.g., LOG_ERROR, LOG_ERR). The identifiers are functionally distinct and no ambiguity exists in practice.
Rule 10.5 The value of an expression should not be cast to an inappropriate essential type. filter_3p3z/COMPONENT_PSC3_P8/filter_3p3z_hw.c Casts to hardware register types are required for configuring the HWFILTER3P3Z peripheral. The register field widths and types are defined by the hardware specification and the PDL register structures. The cast values are range-checked to ensure they fit within the target type.
Rule 17.1 The features of <stdarg.h> shall not be used. cy_pctrl_log.h The logging subsystem uses variadic macros (va_list, va_start, va_end) to provide printf-style formatted log output. This is isolated to the logging module and does not affect the control-path code of the library.
Rule 20.1 The include directives should only be preceded by preprocessor directives or comments. filter_3p3z/COMPONENT_PSC3_P8/filter_3p3z_hw.h The include ordering in the hardware variant header is required to ensure correct resolution of PDL and device-specific types before the block API types are defined.
